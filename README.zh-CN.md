# GPT Image SVG PPT 中文说明

这个 skill 用来把“用户提供的文档和要求”转成一套风格统一的 PPT。

核心路线：

1. 先整体理解文档内容。
2. 先把页数建议和 PPT 内容大纲给用户确认。
3. 按用户确认后的要求规划每页内容。
4. 为整套 PPT 固定整体统一、但不必完全一致的画风。
5. 先用 image2 生成几张完整 PPT 页面例图，给用户确认风格和内容方向。
6. 用户确认例图后，把例图做成参考 PPT 页，让用户检查版式和可编辑性。
7. 再批量生成每页完整 PPT 页面 PNG。
8. 把每页完整 PNG 智能重建成清晰、必要内容可编辑的 SVG/PPT 页面。
9. 把所有 SVG 汇总进 PPTX。
10. 用户要求修改时，只改指定页面，并优先按用户最新要求调整。

## 运行原则

所有内容以用户文件为准。如果文件没有说，或者用户没有明确要求，就不能擅自加入事实、人物、地点、年代、数据、图标含义或画面细节。

这条规则是为了避免“图好看但内容不准”。PPT 中的图像必须服务于文档内容，而不是让图像模型自由发挥。

要求优先级是：

1. 用户最新提出的要求。
2. 用户之前提出且不冲突的要求。
3. 用户提供的源文档。
4. 用户提供的 PPT 模板。
5. skill 默认流程。

如果用户最新要求改变了内容、页数、画风、模板使用方式或修改范围，就要先更新规划，再继续生成或修改。

image2 生成的不是背景图，而是一整页 PPT 画面。也就是说，一页 PNG 应该已经包含本页的整体构图、视觉层级、主要内容块、图像、图解结构和大致文字位置。PPT 里的可编辑文字层主要用于让标题、标签、页码、注释更清晰、更容易修改，而不是把 image2 生成图降级成纯背景。

如果用户提供 PPT 模板，优先在用户模板上制作。需要先读取模板的尺寸、母版、版式、色彩、字体气质、边距和可复用元素，再让 image2 生成适配模板的完整页面。

## 中间文件

skill 要求保留这些中间文件：

- `source-summary.md`：文档理解结果，是后续页面和提示词的依据。
- `outline-proposal.md`：页数建议和 PPT 大纲，用来给用户确认。
- `deck-plan.md`：每页讲什么、画什么、依据来自哪里。
- `style-lock.md`：整套 PPT 的统一画风。
- `samples/`：image2 例图、例图提示词、参考 PPT 页和用户反馈。
- `prompts/`：每页最终发送给图像模型的提示词。
- `png/`：每页 image2 生成的 PNG。
- `svg/`：每页 PNG 转出来的 SVG。
- `qa/`：检查记录。
- `final.pptx`：最终 PPT。

保留这些文件是为了后续可以精确修改，比如只重做第 3 页，而不是整套 PPT 重新开始。

## 流程解释

### 1. 理解文档

先把文档吃透，整理出主题、证据、人物、时间线、用户要求和限制条件。

如果某些内容文档没有支持，要标记为 `NEEDS_USER_APPROVAL`，不能直接写进生图提示词。

如果用户提供了 PPT 模板，这一步也要先分析模板，把模板尺寸、版式和视觉约束记录下来。

### 2. 给出页数建议和大纲

理解完文档后，不要立刻生图。

先输出：

- 建议做多少页
- 为什么建议这个页数
- 每页大概要讲什么
- 是否使用用户模板
- 需要用户确认或修改的地方

必须等用户确认页数和初步大纲，或者明确说“按你的建议继续”，再进入生图阶段。

### 3. 规划 PPT

先决定“每页讲什么”，再决定“每页画什么”。

`deck-plan.md` 里每页都要有：

- 页码和标题
- 本页核心信息
- 要放的内容
- 完整页面画面方向
- 来源依据
- 不允许出现的内容
- 模板使用方式

其中“不允许出现的内容”很重要，用来阻止 image2 自动补充文档没有支持的元素。

### 4. 固定画风

`style-lock.md` 是全套 PPT 画风统一的关键。第一次生图前必须先写好。

它应该包括：

- 构图规则
- 色彩
- 字体气质
- 插画或照片风格
- 光线
- 透视
- 纹理细节
- 图标或图解处理方式
- 用户模板的视觉约束
- 负面提示词
- 每页都要复用的风格后缀

画风只需要整体统一，不要求每一页完全一样。不同页面可以根据内容有不同构图、重点和场景，但要看起来属于同一套 PPT。

### 5. 先生成 image2 例图

在批量生成全套 PPT 前，必须先做例图确认。

先选 2-3 页代表性页面，用 image2 生成完整 PPT 页面 PNG。通常包括：

- 封面或核心观点页
- 普通内容页或图解页
- 有图表、表格、数据的页面，如果文档里有这类内容

把这些例图展示给用户，让用户确认：

- 画风是否合适
- 构图是否合适
- 是否忠于文档
- 是否适配用户模板
- 哪些地方要修改

用户确认例图方向后，才能继续。

### 6. 把例图做成参考 PPT 页

用户确认 image2 例图后，不要立刻批量做全套。

先把确认后的例图按正式流程做成参考 PPT 页，也就是：

- 以 image2 例图为基础。
- 智能重建 SVG/PPT 对象。
- 把重要文字做成可编辑文本框。
- 把图表、表格、数据、坐标轴、图例、标签等做成可编辑对象。
- 复杂背景可以整体保留，只要不影响关键内容编辑。

然后把参考 PPT 页提供给用户检查，让用户提出修改建议。用户确认参考 PPT 页后，再进入全套生成。

### 7. 批量生成完整 PPT 页面 PNG

每页单独生成一张 PNG。提示词必须同时包含：

- 本页内容
- 文档依据
- 视觉目标
- 统一画风
- 用户最新要求
- 模板约束
- 禁止出现的内容

这里生成的是完整 PPT 页面，不是背景图。它应该包含完整构图、主要视觉内容、图解结构和文字区域。

不要让图片模型生成大段文字。标题、标签、注释等需要清晰可改的文字，应在 PPT 里作为可编辑文本叠加或校正。

### 8. PNG 智能重建成 SVG/PPT 页面

PNG 是完整页面中间稿，SVG 是参考 PNG、原文、PPT 大纲和模板后重建出来的完整页面资产。

这里有一个硬性要求：SVG 必须是真矢量，不能是假 SVG。

真 SVG 的意思是文件里主要内容应该由 `<path>`、`<rect>`、`<circle>`、`<polygon>`、`<line>`、`<text>` 等矢量元素组成。不能只是把原来的 PNG/JPEG 用 base64 包进一个 `<image>` 标签里。

最终目标是：在 PowerPoint 里插入 SVG 后，用户执行“转换为形状”时，可以编辑主要信息结构。尤其是文本、图表、表格、数据、坐标轴、图例、标签、关键图解元素必须可编辑。如果 PowerPoint 转换后这些内容还是不可编辑的位图，或者无法拆出有意义的形状，这个 SVG/PPT 页就不合格。

但不是要求 image2 生成的所有视觉都变成 PPT 原生要素。复杂背景、氛围图、装饰纹理、非关键插画细节可以整体保留或作为锁定背景层，只要清晰度足够，且不影响文本、图表、表格、数据等关键内容的编辑。

也不要把每个色块、阴影、纹理、抗锯齿边缘都机械拆成大量矢量路径。那样虽然“真矢量”，但 PPT 会很卡，也很难编辑。正确做法是利用 GPT/视觉理解做智能重建：

- 参考 image2 生成的完整页面效果。
- 参考源文档确认内容。
- 参考 `deck-plan.md` 确认每页结构。
- 参考 PPT 模板和文本框位置。
- 把主要版块、图形、图标、箭头、时间线、图表、人物/物体轮廓重建为清晰矢量对象。
- 把重要文字优先做成 PPT 可编辑文本框，而不是描成不可编辑的文字轮廓。
- 把图表、表格、数据标签、坐标轴、图例等优先做成可编辑元素。
- 允许复杂背景整体保留，以保持清晰度和性能。

如果转出来的是假 SVG，要重新处理：

- 让 image2 生成更扁平、更少噪点、更适合语义重建的页面。
- 换用能输出真矢量或语义 SVG 的工具。
- 利用 GPT/视觉理解重新生成结构化 SVG。
- 对图表、时间线、文字、箭头等关键内容，手工重建为 SVG 或 PPT 形状/文本框。

为了 SVG 效果好，前面的生图风格最好偏：

- 扁平
- 清晰
- 低噪点
- 色块明确
- 边缘干净

复杂写实照片转成 SVG 往往体积大、难编辑、效果不稳定。

### 9. 汇总 PPT

每个 SVG 对应一页 PPT。SVG full-bleed 放到对应页面。

如果用户提供模板，优先把 SVG 放进用户模板对应页面，而不是新建空白模板。

标题、标签、页码、注释等需要清晰可改的文字，优先用 PPT 文本层，不要依赖图片里的文字。

### 10. 交付前检查

交付前必须检查：

- 是否符合用户确认过的页数和大纲
- 内容是否忠于文档
- 页序是否正确
- 是否适配用户模板
- 画风是否整体统一
- 是否有跑题图像
- 是否有模型编造的文字或元素
- 是否有裁切、遮挡、空白页
- SVG 是否能在 PPT 中正常显示
- SVG 是否是真矢量，而不是嵌入位图
- PowerPoint 中“转换为形状”后是否能编辑主要元素
- 是否避免了把每个细碎色块都拆成不可用的大量路径
- 重要文字、图表、表格、数据是否可编辑
- 复杂背景是否保持清晰

## 修改规则

用户要求修改时，不要默认重做整套 PPT。

默认做法：

1. 重新读取 `deck-plan.md`、`style-lock.md` 和对应页提示词。
2. 优先执行用户最新提出的修改要求。
3. 只改用户点名的页面。
4. 保持整套 PPT 视觉上协调，但如果用户最新要求改变某页风格或内容，要以用户最新要求为准。
5. 如果修改涉及事实内容，必须回到文档找依据，或者向用户确认。
6. 新提示词和图片用版本号保存，比如 `slide-03-v2`。
7. 替换对应 SVG 后重新汇总 PPT。

如果用户明确要求全局换风格，才更新 `style-lock.md` 并重做相关页面。

## 提示词约束

每页提示词必须有固定结构，确保 image2 知道：

- 要画什么
- 为什么这样画
- 哪些不能画
- 如何保持整体统一画风
- 是否要适配用户模板

如果文档只说“1850 年后棉纺织出口上升”，就不能擅自提示“蒸汽工厂内部和机器轰鸣”，除非文档支持或用户明确批准。

## 最终交付

最终交付不能只有 PPT。还要保留：

- 规划文件
- 风格锁
- 提示词
- PNG
- SVG

这样后续才能精确修改和追踪每页来源。
